# 설계 방향 제안 (래더 학습 결과 기반)

> [`ladder-knowledge-base.md`](ladder-knowledge-base.md)의 학습을 우리 코드(`app/`)에 어떻게 반영할지 구체화한다.
> 각 제안은 **현재 코드 → 제안 → 근거(학습)** 형식. 우선순위와 즉시 착수 항목을 말미에 둔다.

---

## 0. 핵심 설계 원칙 (학습에서 도출)

1. **벤더중립 IR → 어댑터 렌더.** 주소 모델이 벤더마다 근본 호환불가(8진/16진/byte.bit/태그)다. 파이프라인은 **벤더중립 중간표현**을 만들고, 마지막에 **벤더 프로파일**이 디바이스·명령어·주소를 입힌다.
2. **자유생성 대신 패턴 조립.** 연구·실무 공통: **LD 자유생성은 단순 케이스도 난제**다. 현업은 이름붙은 패턴(자기유지·인터록·first-out·스텝)으로 사고한다. → LLM에 래더를 자유작성시키지 말고, **결정론 패턴 템플릿**을 I/O로 파라미터화해 조립. LLM은 "요구→패턴 선택+파라미터 추출"만.
3. **검증은 안전을 보장하지 않는다.** 소프트 인터록 ≠ 안전기능. E-stop은 하드와이어. 도구는 "논리 보조"로 명확히 포지셔닝.
4. **실무 입력은 물리 접점 언어.** 한국 사용자는 불리언이 아니라 "누르면 붙고 떼면 떨어진다"로 말한다. 분석기는 릴레이-시퀀스 멘탈모델 전제.
5. **결정론 우선 유지(기존 강점).** 이중코일 기계제거·검증 게이트·검증통과분만 학습 — 그대로 간다.

## 1. 벤더 프로파일 추상화 (아키텍처 최우선)

**현재**: `app/memory_map.py`에 LS식 디바이스·자릿수 하드코딩.
```python
_DIGIT_WIDTH = {DeviceClass.P:4, DeviceClass.M:4, ... DeviceClass.D:5}
def _format(dc, idx): return f"{dc.value}{idx:0{width}d}"
```
`app/models.py`의 `DeviceClass` enum도 LS 고정(P/M/T/C/L/K/D).

**제안**: `app/vendors/` 신설 — `VendorProfile` 프로토콜.
```python
class VendorProfile(Protocol):
    name: str                                  # "LS_XGK" | "MITSUBISHI_FX" | "SIEMENS_S7" ...
    def device_for(self, role: DeviceRole) -> str        # 입력/출력/내부/타이머/...
    def format_address(self, role, index) -> str         # 진법·자릿수 (16진/8진/byte.bit/태그)
    def mnemonic(self, op: LadderOp) -> str              # NO/NC/coil/set/oneshot/timer
    def timer_is_countdown(self) -> bool                 # Omron TIM = 감산
```
- 기본 프로파일 = **LS_XGK**(한국 타깃, 기존 동작 보존). 이어서 MITSUBISHI_FX, SIEMENS_S7.
- `DeviceClass`를 벤더 무관 **`DeviceRole`**(INPUT/OUTPUT/INTERNAL/KEEP/TIMER/COUNTER/STEP/DATA)로 일반화하고, 프로파일이 role→실제 디바이스 문자로 변환.
- `memory_map`의 `_format`/`build_from_spec`이 프로파일을 받게 변경(기본값 LS로 하위호환).

**근거**: KB §2-6 — 상승엣지 원샷만 해도 OUTP/PLS/P/DIFU/OSR로 전부 다름; 주소 8진(FX)/16진(LS)/byte.bit(S7)/태그(AB) 호환불가. 어댑터 없이는 한 벤더에만 묶임.
**완료기준**: LS 프로파일로 기존 12 골든셋 회귀 통과 + 미쓰비시 프로파일로 같은 spec이 X/Y 8진 주소로 렌더되는 테스트.

## 2. RAG 코퍼스 = 실제 벤더 명령어셋 (ROI 최고)

**현재**: `data/instructions.jsonl`이 더미 스텁(NO_CONTACT/NC_CONTACT/COIL 일반론).
**제안**: KB §2의 **벤더별 디바이스·명령어·주소 사실**로 교체·확장. 항목 스키마에 `vendor` 필드 추가, 명령어마다 {벤더, 디바이스 제약, 진법, 시간베이스, 엣지의미, 예제, 출처}.
- 청크 단위 = 명령어 1개(예: 미쓰비시 PLS, LS OUTP, Omron TIM 감산, Siemens TON 인스턴스DB).
- `app/rag.py`의 검색을 벤더 필터 가능하게.

**근거**: arxiv 2511.09122 — 순정 LLM은 미쓰비시 `RTRIG_P` 상승엣지·외부라벨 선언 등 벤더의미를 모름. **RAG 주입으로 GX Works3 ST 컴파일 38%→73~87%.** 우리 RAG는 지금 더미라 이 레버를 못 쓰고 있음.
**완료기준**: "타이머"/"상승엣지"/"정역" 질의에 벤더별 정확한 명령어 청크가 top-k 등장. 골든셋 생성 시 벤더 프로파일에 맞는 명령어만 인용.

## 3. 패턴 라이브러리 — 이름붙은 결정론 템플릿 (신뢰도 핵심)

**현재**: `app/transpiler.py`가 ST→래더(Sum-of-Products) 결정론 변환. 패턴 개념 없음. 시퀀싱은 조합로직 위주.
**제안**: `app/patterns/` 신설 — 파라미터화된 래더 패턴 템플릿 카탈로그.
- 1급 패턴(KB §1,§4): `seal_in`(자기유지), `interlock`(상호배제), `start_stop`(stop-dominant, NC stop), `jog`, `flasher`, `star_delta`, `debounce`(TON+TOF), `first_out_alarm`, `mode`(auto/manual/jog), `step_sequencer`(§4), `lead_lag`.
- 각 템플릿: `build(params) -> list[LadderRung]` (결정론, 테스트 가능, 이중코일 0 보장).
- 파이프라인 변경: analyst가 요구→**패턴 식별 + I/O 바인딩** 산출 → architect가 **템플릿 조립**(자유 ST 생성 최소화) → verifier/renderer 그대로.

**근거**: KB §1 실무자는 패턴 카탈로그(seal-in/fault coil/debounce/flasher/step/mode)로 사고; 연구상 LD 자유생성은 미해결. 템플릿 조립이 환각을 구조적으로 차단(이중코일 후처리와 같은 사상).
**완료기준**: 11개 패턴 각각 단위테스트(파라미터→기대 rung 트리, 이중코일0). 골든셋 ㉗first-out·⑧star-delta가 템플릿으로 재현.

## 4. 스텝/상태 시퀀서 모델 (PLAN H2 연계)

**현재**: 시퀀싱이 Sum-of-Products 조합로직. 자기유지 체인 방식의 한계(스파게티).
**제안**: **정수 스텝 레지스터** 시퀀서를 1급 모델로. 상태 동작은 `EQU(step=N)`로 게이팅, 전이는 증가/`MOV`. 벤더 프로파일의 STEP role(LS **S**, 미쓰비시 **S**)에 매핑. SFC형 spec → 스텝 래더 결정론 변환.

**근거**: KB §1(레벨8), §3(한국 강사들이 자기유지 체인 대신 스텝제어 권장), §4(신호등·배칭이 스텝/타이머ACC비교 구조). "한 번에 한 상태만 활성 → 진단 쉬움".
**완료기준**: 신호등(6스텝)·배칭(충진→교반→배출)이 스텝 시퀀서로 생성, 검증 통과.

## 5. 분석 에이전트 = 릴레이-시퀀스 멘탈모델

**현재**: `app/prompts.py`의 analyst가 I/O·조건 추출(일반적).
**제안**: 프롬프트·spec을 **자기유지 3원소(기동 a접점 / 자기유지 / 해제 b접점)** 중심으로. 한국어 물리-접점 표현("누르면 붙고") → 패턴 매핑 규칙. KB §3-2 용어표를 분석기 컨텍스트로 주입. "정역/비상정지/Y-Δ" 같은 관용 요구를 패턴 ID로 직결.

**근거**: KB §3-1 — 한·일 사용자는 래더를 그려진 릴레이로 사고, 입력이 불리언이 아니라 물리동작. 안전은 b접점에 산다(자기유지 경로 차단).
**완료기준**: "버튼 누르면 모터 돌고 정지누르면 선다" → seal_in 패턴+stop NC 자동 식별 테스트.

## 6. 실사용 출력 — PLCopen XML 익스포트 + 컴파일 게이트

**현재**: 출력은 래더 JSON(`LadderProgram`) — 실무 IDE가 못 읽음.
**제안**:
- `app/export/plcopen_xml.py` — `LadderProgram`→**PLCopen XML(IEC 61131-10)**. OpenPLC Editor/CODESYS로 라운드트립.
- 평가 하니스에 **matiec 컴파일 게이트** 추가(생성 ST를 matiec로 문법검증). 가능하면 nuXmv 인터락 모델체크.
- 벤더 어댑터 2순위: L5X(Rockwell), TIA Openness XML(Siemens).

**근거**: KB §5 — PLCopen XML이 유일한 중립 표준, OpenPLC가 직접 임포트. matiec+nuXmv = LLM4PLC 검증스택, 오픈소스·API키 불필요(우리 CI 규칙 부합).
**완료기준**: 골든 ST가 matiec 문법 게이트 통과; 생성 XML이 OpenPLC Editor에서 로드(수동 1케이스).

## 7. 골든셋 현실화 (교과서 → 실무 ~30종)

**현재**: 12개 전부 단순 예제(`tests/fixtures/golden/`).
**제안**: KB §4의 ~30 케이스로 확장, **커리큘럼 레벨 태깅**(§1). 각 케이스에 `{자연어, spec, 기대 패턴ID, 기대 ST, 금지조건(이중코일0/인터락0), 벤더, 출처}`. 평가지표에 I/O 커버리지·패턴 정확도 추가.

**근거**: KB §1,§4. 현 골든셋은 신뢰의 게이트지만 현실성이 낮아 "실사용 무리"(사용자 지적).
**완료기준**: 30 케이스 이중코일0·인터락0 유지, 패턴 매핑 정확도 리포트.

## 8. 안전 경계 명문화 (전반)

- README/출력/UI에 고지: **"본 도구의 검증은 논리 보조이며, 안전기능(E-stop·가드·정역 전원차단)은 ISO 13849/IEC 62061 하드와이어 안전회로로 별도 구현해야 한다. 인터록 검증은 안전 인증이 아니다."**
- first-out/E-stop 패턴 출력에 "하드웨어 안전릴레이 권장" 주석 자동 첨부.

**근거**: 실무조사 보고서 §4 + KB §3-1. 벤더 코파일럿들도 SIL2+ 안전로직 자율생성 안 함.

## 부록: 한 장 요약
- **무엇을 배웠나**: 래더는 벤더마다 주소·명령어가 근본 다르고(중립IR+어댑터 필요), 실무는 이름붙은 패턴으로 짜며(자유생성 아님), 한국 사용자는 물리접점 언어로 말하고, 출력은 PLCopen XML이라야 실IDE에 들어가며, 안전은 래더 밖 하드웨어에 있다.
- **어디로 갈까**: 결정론 강점(이중코일·검증) 유지 + ① 벤더 프로파일 ② 패턴 템플릿 조립 ③ 실벤더 RAG ④ 스텝 시퀀서 ⑤ 릴레이-시퀀스 분석기 ⑥ PLCopen XML/matiec ⑦ 현실 골든셋 ⑧ 안전 고지.
- **지금 당장**: Phase K(실벤더 RAG + 현실 골든셋 + 안전 고지) — 키 불필요·결정론·즉효.

### 래더 생성 난제 — 해법 요약
연구가 "LD 자유생성 미해결"이라 한 문제를 **LLM을 래더 로직에서 제거**해 우회한다:
**자연어 →(LLM)→ 명세 →(결정론 합성)→ ST →(Z3 검증 + matiec 컴파일)→ 래더 →(벤더 에미터/PLCopen)→ 실코드.**
골든 20개 중 19개가 LLM 없이 완전 결정론 생성(이중코일0·인터락0·검증통과). 조합출력(알람 경음기 등)만 LLM/패턴 폴백.
