## LS ELECTRIC (XGT/XGK/XGB)

### 통신

#### `CNET_NO_RESP` — Cnet(시리얼) 통신 무응답/타임아웃

**심각도**: 🟡 경고 · **시리즈**: XGB/XGT Cnet

- **원인**: 국번·보레이트·패리티 불일치, RS-485 A/B 극성 반전, 종단저항 누락.
- **조치(실무)**: 양단 파라미터(국번/속도/패리티/정지비트)를 표로 만들어 1:1 대조. 485 라인은 A-A/B-B 결선·말단 종단저항(120Ω) 확인. 프레임 모니터 기능으로 요청/응답 프레임을 캡처하면 어느 쪽이 침묵하는지 즉시 갈린다.
- **근거/출처**: [XGK 초급 교재](https://sol.ls-electric.com/uploads/document/16572866006190/XGK+초급(V21).pdf)
- **키워드**: Cnet, RS485, 통신, 타임아웃, 국번

#### `COMM_ERR` — XGB 통신 오류

**심각도**: 🟡 경고 · **시리즈**: XGB

- **원인**: 내장 시리얼 또는 Cnet 통신 포트의 응답 불량.
- **조치(실무)**: 통신 파라미터(보레이트, 프로토콜) 재확인, 케이블 및 종단저항 점검.
- **키워드**: XGB, communication, serial, Cnet, LS ELECTRIC

#### `FENET_IP_DUP` — FEnet(이더넷) 통신 두절 — IP 충돌/링크 다운

**심각도**: 🟡 경고 · **시리즈**: XGT FEnet

- **원인**: 동일 네트워크에 IP 중복, 스위치 포트 불량, 케이블 단선.
- **조치(실무)**: 모듈 LINK/ACT LED 로 물리 링크부터 확인 → arp/ping 으로 IP 중복 검사 (중복 시 한쪽 변경). 채널 점유 한계(동시 접속 수) 초과도 두절처럼 보이므로 접속 클라이언트 수를 확인.
- **근거/출처**: [XGK 초급 교재](https://sol.ls-electric.com/uploads/document/16572866006190/XGK+초급(V21).pdf)
- **키워드**: FEnet, 이더넷, IP 충돌, LINK

### I/O·모듈

#### `IO_CONF_ERR` — XGK I/O 구성 오류

**심각도**: 🔴 정지급 · **시리즈**: XGK

- **원인**: XGK 기저부에 장착된 모듈이 XG5000 프로젝트 I/O 파라미터와 불일치.
- **조치(실무)**: XG5000 I/O 파라미터를 실제 장착 모듈에 맞게 수정 후 재다운로드.
- **키워드**: XGK, I/O, configuration, LS ELECTRIC, XG5000

#### `IO_DETACH` — 운전 중 모듈 착탈 감지 — CPU 정지

**심각도**: 🔴 정지급 · **시리즈**: XGT 공통

- **원인**: 운전 중 I/O 모듈이 빠졌거나(진동·베이스 록 풀림) 접점 산화로 착탈로 오인식.
- **조치(실무)**: 모듈 록 레버·베이스 고정 나사 점검 후 재장착. 진동 심한 설비는 베이스 고정 보강. 재발 시 해당 슬롯 커넥터 청소(접점 세정제) 후 이력 클리어.
- **근거/출처**: [XGK 초급 교재](https://sol.ls-electric.com/uploads/document/16572866006190/XGK+초급(V21).pdf)
- **키워드**: 모듈 착탈, 탈락, 베이스, 정지

#### `IO_PARAM_MISMATCH` — I/O 파라미터 불일치 — 장착 모듈과 설정 모듈이 다름

**심각도**: 🔴 정지급 · **시리즈**: XGK

- **원인**: 슬롯에 장착된 모듈 종류가 I/O 파라미터 설정과 다르거나, 모듈 교체 후 파라미터를 갱신하지 않음.
- **조치(실무)**: XG5000 [I/O 파라미터]에서 슬롯별 설정과 실장 모듈을 대조. 모듈을 바꿨다면 파라미터 재작성 후 쓰기. 베이스 증설 케이블 접촉 불량도 동일 증상을 내므로 케이블 재삽입 확인.
- **근거/출처**: [XGK 초급 교재](https://sol.ls-electric.com/uploads/document/16572866006190/XGK+초급(V21).pdf)
- **키워드**: I/O, 파라미터, 모듈 불일치, 슬롯

#### `_IO_RWER` — 모듈 입출력(읽기/쓰기) 에러 플래그 — 모듈 액세스 실패

**심각도**: 🔴 정지급 · **시리즈**: XGI/XGK

- **원인**: 모듈 내부 고장, 베이스 커넥터 접촉 불량, 강한 노이즈로 버스 액세스가 실패. 특정 슬롯에서 반복되면 그 모듈/슬롯 하드웨어가 유력.
- **조치(실무)**: 에러 이력에서 슬롯 위치 특정 → 전원 차단 후 모듈 재장착(커넥터 청소). 재발 시 정상 슬롯과 모듈을 맞바꿔 '모듈 따라가는지/슬롯에 남는지'로 고장 부위를 갈라낸다 — 모듈이면 교체, 슬롯이면 베이스 교체.
- **근거/출처**: [XGI CPU 매뉴얼 플래그 일람](https://ssq.ls-electric.com/uploads/document/16411831069720/XGI-CPU_Manual_202012_V2.1_EN.pdf)
- **키워드**: _IO_RWER, 모듈 액세스, 베이스, 교차 시험

#### `_IO_TYER` — 모듈 타입 불일치 플래그 — 설정 모듈과 실장 모듈이 다름

**심각도**: 🔴 정지급 · **시리즈**: XGI/XGK

- **원인**: I/O 파라미터에 등록된 모듈 형식과 슬롯의 실제 모듈이 다르거나, 호환되지 않는 후속 모델로 교체한 뒤 파라미터를 갱신하지 않음.
- **조치(실무)**: XG5000 에러 이력에서 _IO_TYER 와 함께 기록된 슬롯 번호를 확인 → 해당 슬롯의 실장 모듈 형명과 I/O 파라미터를 대조해 한쪽을 맞춘다. 후속 호환 모델 교체 시에도 형명이 다르면 타입 에러가 나므로 파라미터를 반드시 재작성.
- **근거/출처**: [XGI CPU 매뉴얼(공식 공개 PDF) 플래그 일람](https://ssq.ls-electric.com/uploads/document/16411831069720/XGI-CPU_Manual_202012_V2.1_EN.pdf)
- **키워드**: _IO_TYER, 타입 불일치, F 플래그, 슬롯

#### `XBF-AD-WIRING` — 아날로그 입력 모듈 값 이상/고정 — 외부 결선·설정 불일치

**심각도**: 🟡 경고 · **시리즈**: XGB XBF-AD

- **원인**: 전류/전압 입력 모드와 실제 센서 출력 불일치, 미사용 채널 개방, 실드 미접지로 값 떨림.
- **조치(실무)**: 모듈 운전 파라미터(전압/전류, 범위)와 센서 사양을 1:1 대조 → 미사용 채널은 '사용 안 함'으로 막아 진단 알람을 없앤다. 값 떨림은 실드 편측 접지·동력선 분리 배선으로 잡는 것이 정석.
- **근거/출처**: [XBF-AD04A 사용설명서(공식 공개 PDF)](https://ssq.ls-electric.com/uploads/document/16401485635220/XBF-AD04A_T16_Manual_V2.2_202110_KR.pdf.pdf)
- **키워드**: XBF, 아날로그, 결선, 채널 설정

#### `_FUSE_ER` — 퓨즈 단선 경고 플래그 — 출력 모듈 퓨즈 끊어짐

**심각도**: 🟡 경고 · **시리즈**: XGI/XGK

- **원인**: 출력 모듈 퓨즈가 부하 단락/과부하로 용단 — 해당 모듈 출력만 죽고 CPU 운전은 지속되는 경우가 많아 발견이 늦다.
- **조치(실무)**: 퓨즈 교체 전에 반드시 원인(출력단 단락·솔레노이드 소손·배선 피복 손상)을 먼저 제거 — 원인 없이 갈면 또 끊어진다. 교체 후 _FUSE_ER 플래그를 HMI 알람에 매핑해 다음번엔 즉시 인지되게 한다.
- **근거/출처**: [XGI CPU 매뉴얼 플래그 일람](https://ssq.ls-electric.com/uploads/document/16411831069720/XGI-CPU_Manual_202012_V2.1_EN.pdf)
- **키워드**: _FUSE_ER, 퓨즈, 출력 모듈, 단락

### 전원/배터리

#### `BAT-LED` — BAT LED 점등 — 백업 배터리 전압 저하

**심각도**: 🟡 경고 · **시리즈**: XGT 공통

- **원인**: CPU 백업 배터리(리튬) 수명 종료 임박 — 정전 시 래치 데이터·시계 소실 위험.
- **조치(실무)**: 통전 상태에서 교체하면 데이터가 보존된다(정전 교체 시 래치 영역 소실 위험 — LS 공식 Q&A 절차). 교체 전 XG5000 으로 프로젝트/데이터 백업 필수. 기종별 배터리 형번은 공식 '기종별 배터리' 문서로 확인 후 정품 사용.
- **근거/출처**: [LS ssq Q&A: CPU 백업 배터리 교체방법](https://ssq.ls-electric.com/kr/ko/community/qna/document/25825)
- **키워드**: 배터리, BAT, 래치, 백업, 교체

#### `BATT_ERR` — XGK 배터리 이상

**심각도**: 🟡 경고 · **시리즈**: XGK

- **원인**: XGK CPU 모듈 배터리 전압 저하 또는 배터리 미장착.
- **조치(실무)**: 지정 규격의 배터리로 교체 후 메모리 데이터 재확인.
- **키워드**: XGK, battery, BATT, LS ELECTRIC, RAM backup

### 프로그램/연산

#### `PROG_ERR` — XGK 프로그램 오류

**심각도**: 🔴 정지급 · **시리즈**: XGK

- **원인**: 래더 프로그램 내 문법 오류, 잘못된 명령어 사용, 또는 END 명령 누락.
- **조치(실무)**: XG5000으로 오류 위치 확인 후 수정 및 재다운로드.
- **키워드**: XGK, program, error, ladder, LS ELECTRIC

#### `_CODE_ER` — 명령어 코드 에러 플래그 — 해석 불가 코드 실행 시도

**심각도**: 🔴 정지급 · **시리즈**: XGI/XGK

- **원인**: 다운로드 중 케이블 분리 등으로 프로그램 메모리가 부분 손상됐거나, CPU OS 버전이 지원하지 않는 신규 명령을 사용.
- **조치(실무)**: XG5000 으로 프로젝트를 다시 전송(쓰기 후 베리파이까지) → 재발 시 CPU OS 버전과 XG5000 버전 호환표 확인 후 OS 업데이트. 산발 재발이면 전원· 노이즈 계통(접지/실드) 점검으로 메모리 손상 원인을 제거.
- **근거/출처**: [XGI CPU 매뉴얼 플래그 일람](https://ssq.ls-electric.com/uploads/document/16411831069720/XGI-CPU_Manual_202012_V2.1_EN.pdf)
- **키워드**: _CODE_ER, 명령어 코드, OS 버전, 베리파이

#### `XGB-HSC-PARAM` — 고속카운터 카운트 누락/불가 — 파라미터·입력 사양 불일치

**심각도**: 🟡 경고 · **시리즈**: XGB HSC

- **원인**: 엔코더 출력(오픈컬렉터/라인드라이버)과 모듈 입력 사양 불일치, 1상/2상 모드 설정 오류, 입력 필터가 펄스 폭보다 길어 카운트 누락.
- **조치(실무)**: 엔코더 사양서와 모듈 입력 사양(전압·최대 주파수)을 먼저 대조 → 2상 엔코더는 A/B 위상 결선과 모드 일치 확인 → 고주파에서 누락되면 입력 필터 시간을 펄스 폭 이하로 낮춘다. 설정 후 저속 수동 회전으로 1:1 카운트 검증.
- **근거/출처**: [LS ssq — XGB 고속카운터 사용설명서 문서 페이지](https://ssq.ls-electric.com/kr/ko/product/document/1408)
- **키워드**: 고속카운터, HSC, 엔코더, 입력 필터

### CPU/시스템

#### `CPU_ERR` — XGK CPU 하드웨어/자기진단 오류

**심각도**: 🔴 정지급 · **시리즈**: XGK

- **원인**: XGK CPU 모듈 내부 자기진단에서 하드웨어 이상 감지.
- **조치(실무)**: CPU 모듈 교체 및 LS ELECTRIC 기술지원 문의.
- **키워드**: XGK, CPU, error, self-diagnostic, LS ELECTRIC

#### `CPU_ERR` — XGB CPU 이상

**심각도**: 🔴 정지급 · **시리즈**: XGB

- **원인**: XGB CPU 자기진단 실패 또는 내부 하드웨어 불량.
- **조치(실무)**: CPU 모듈 전원 재투입 후 지속 시 모듈 교체.
- **키워드**: XGB, CPU, error, LS ELECTRIC

#### `ERR-LED-ON` — ERR LED 점등 — CPU 자기진단 에러(운전 정지급)

**심각도**: 🔴 정지급 · **시리즈**: XGT 공통

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: CJ/CALL 이 가리키는 P 라벨이 삭제됐거나 번호가 바뀜(부분 수정 시 흔함).
- **조치(실무)**: 크로스 레퍼런스로 모든 CJ/CALL→P 짝을 검사. 라벨 번호를 직접 치지 말고 GX Works 의 라벨 점프 기능으로 따라가며 빠진 짝을 복원.
- **근거/출처**: [FX3U 에러표](https://www.manualslib.com/manual/1234627/Mitsubishi-Fx3u-Series.html?page=261)
- **키워드**: 6701, CJ, CALL, P 라벨

#### `6702` — CALL 중첩 한도 초과 — 서브루틴 6단 이상 호출

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: CALL 의 중첩이 허용 단수(5단)를 넘음 — 서브루틴 안에서 또 CALL 하는 구조가 누적된 경우. 조건 분기에 따라 런타임에만 터지기도 한다.
- **조치(실무)**: 크로스 레퍼런스로 CALL 호출 트리를 그려 최심 깊이를 센다 → 5단 이내로 평탄화(말단 서브루틴을 상위로 인라인하거나 플래그+메인 분기로 변경). 재귀성 호출(자기 자신/순환 CALL)은 FX 에서 금물이므로 구조 자체를 제거.
- **근거/출처**: [FX3U 프로그래밍 매뉴얼 에러표(공개 열람)](https://www.manualslib.com/manual/1304866/Mitsubishi-Electric-Melsec-Fx3u.html?page=942)
- **키워드**: 6702, CALL, 중첩, 서브루틴, P 라벨

#### `6703` — 인터럽트 중첩 한도 초과 — 3단 이상 중첩

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: 인터럽트 루틴 실행 중 다른 인터럽트가 연쇄로 끼어들어 허용 단수(2단)를 초과 — 고속 펄스 입력이 몰리는 설비에서 전형적.
- **조치(실무)**: 인터럽트 루틴은 '플래그만 세우고 즉시 IRET' 수준으로 최소화하고 실처리는 메인 스캔으로 이관. 동시에 쓰는 인터럽트 수 자체를 줄이고, 필요 구간에서만 EI/DI 로 허가 범위를 좁힌다.
- **근거/출처**: [FX3U 프로그래밍 매뉴얼 에러표](https://www.manualslib.com/manual/1304866/Mitsubishi-Electric-Melsec-Fx3u.html?page=942)
- **키워드**: 6703, 인터럽트, 중첩, EI, DI

#### `6704` — FOR-NEXT 중첩 한도 초과 — 6단 이상

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: FOR~NEXT 루프가 허용 단수(5단)를 넘게 중첩 — 복붙으로 루프 안에 루프를 계속 넣다 보면 도달한다.
- **조치(실무)**: 중첩 루프를 1~2중으로 풀어 쓰되, 다차원 순회는 인덱스 산술(행×폭+ 열)로 1중 루프화. 루프가 깊을수록 스캔타임도 같이 폭증하므로 6105(워치독)와 함께 오는지 D8012 를 확인.
- **근거/출처**: [FX3U 프로그래밍 매뉴얼 에러표](https://www.manualslib.com/manual/1304866/Mitsubishi-Electric-Melsec-Fx3u.html?page=942)
- **키워드**: 6704, FOR, NEXT, 중첩, 루프

#### `6705` — 명령 피연산자 부적합 — 대상 디바이스 타입 불일치

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: 응용명령에 허용되지 않는 디바이스 종류를 지정(예: 워드 자리에 비트).
- **조치(실무)**: 에러 스텝(D8069)으로 해당 명령을 특정 → 매뉴얼의 그 명령 '사용 가능 디바이스' 표와 대조해 교정. 라이브러리 복붙 코드에서 자주 발생.
- **근거/출처**: [FX3U 에러표](https://www.manualslib.com/manual/1234627/Mitsubishi-Fx3u-Series.html?page=261)
- **키워드**: 6705, 피연산자, 디바이스 타입

#### `6706` — 디바이스 범위 초과 — 인덱스(V/Z) 수식이 범위 밖을 가리킴

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: V/Z 인덱스 값이 커져 D/M 의 실제 범위를 벗어난 접근(런타임에 발생, 정지 상태에선 안 보임).
- **조치(실무)**: 실무 검증된 절차(MrPLC 포럼 사례): 에러 시점의 V/Z 를 HMI/래더로 래치해 두고, 인덱스 사용처마다 상한 비교(CMP)로 가드 — '인덱스 가드 없는 간접참조는 시한폭탄'. 에러 스텝(D8069)으로 위치 특정 후 가드 삽입.
- **근거/출처**: [MrPLC 포럼 — FX2N Error 6706 해결 스레드](https://mrplc.com/forums/topic/32854-error-6706-mitsubishi-fx2n/)
- **키워드**: 6706, 인덱스, V, Z, 범위 초과, 간접참조

#### `6707` — 파일 레지스터 미설정 접근

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: 파라미터에서 파일 레지스터 영역을 확보하지 않고 R/ER 영역을 접근.
- **조치(실무)**: PLC 파라미터의 메모리 용량 설정에서 파일 레지스터 블록 수를 확보 → 쓰기 → 전원 재투입. 다른 현장 프로그램 이식 때 빠뜨리기 쉬운 항목.
- **근거/출처**: [FX3U 에러표](https://www.manualslib.com/manual/1234627/Mitsubishi-Fx3u-Series.html?page=261)
- **키워드**: 6707, 파일 레지스터, 파라미터

#### `6708` — FROM/TO 명령 에러 — 특수 유닛 버퍼메모리(BFM) 액세스 실패

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: 유닛 번호/BFM 번호가 실장 구성과 안 맞거나, 특수 유닛이 응답하지 않음. 간헐 발생이면 증설 케이블 접촉·노이즈로 액세스가 무작위 실패하는 사례가 보고됨.
- **조치(실무)**: 에러 스텝(D8069)의 FROM/TO 에서 유닛 번호(K0부터 좌→우)와 BFM 번호를 그 유닛 매뉴얼과 대조. 간헐성이면 증설 케이블 재삽입·노이즈 대책 후 M8067(연산 에러 래치)로 재발을 감시한다 — 매번 같은 스텝이면 프로그램, 무작위 스텝이면 하드웨어 쪽이 유력.
- **근거/출처**: [IndustrialMonitorDirect KB — FX 간헐 6708 사례](https://industrialmonitordirect.com/blogs/knowledgebase/mitsubishi-fx-series-plc-intermittent-cpu-e-error-and-operation-error-6708-troubleshooting)
- **키워드**: 6708, FROM, TO, BFM, 특수 유닛

#### `6709` — 기타 연산 구조 에러 — IRET/SRET 누락·FOR-NEXT 짝 이상

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: 인터럽트/서브루틴의 복귀 명령(IRET/SRET) 누락, FOR-NEXT 짝 불일치, 점프로 루프 중간에 뛰어드는 비정상 흐름.
- **조치(실무)**: P 라벨 이후 블록마다 SRET, I 포인터 루틴마다 IRET 가 닫히는지 전수 점검 → FOR/NEXT 개수 일치 확인 → CJ 가 FOR-NEXT 내부로 점프해 들어가는 경로가 없는지 흐름을 따라간다. 부분 복붙 수정 직후에 가장 잘 터진다.
- **근거/출처**: [FX3U 프로그래밍 매뉴얼 에러표](https://www.manualslib.com/manual/1304866/Mitsubishi-Electric-Melsec-Fx3u.html?page=942)
- **키워드**: 6709, IRET, SRET, FOR-NEXT, 점프

#### `PARAM_ERR` — iQ-R 파라미터 오류

**심각도**: 🔴 정지급 · **시리즈**: iQ-R

- **원인**: CPU 파라미터 설정값이 유효 범위를 벗어나거나 파라미터 파일 손상.
- **조치(실무)**: GX Works3로 파라미터 재확인 및 올바른 값으로 수정 후 재쓰기.

#### `PROG_ERR` — FX 프로그램/명령어 오류

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: FX CPU가 실행할 수 없는 명령어 또는 피연산자 범위 초과.
- **조치(실무)**: GX Works2로 오류 스텝 확인 후 명령어 수정 및 재전송.

### CPU/시스템

#### `6101` — RAM 에러 — CPU 메모리 자기진단 실패

**심각도**: 🔴 정지급 · **시리즈**: FX

- **원인**: CPU RAM 불량 또는 강한 노이즈/순간정전으로 메모리 파손.
- **조치(실무)**: 전원 재투입 후 재발 여부 확인 → 재발 시 프로그램 백업 후 메모리 올클리어·재전송. 그래도 반복되면 CPU 하드웨어 불량 — 본체 교체가 정답이며 노이즈 대책(전원 라인 필터·접지 분리)을 함께 검토.

#### `D8060-D8067` — 에러코드 저장 체계 — 특수 레지스터 D8060~D8067/D8438

**심각도**: 🔵 절차/지식 · **시리즈**: FX 공통

- **원인**: (체계 설명) 에러 종류별로 다른 D 레지스터에 코드가 저장되고 대응 M8060~M8067 플래그가 선다.
- **조치(실무)**: GX Works2/3 [진단]-[PLC 진단]으로 현재/이력 에러를 읽는 것이 첫 단계. 래더에서 D8060~D8067 을 HMI 로 상시 표시해 두면 현장에서 코드부터 부른다 — 원격 지원이 한 단계 빨라지는 실무 팁.
- **근거/출처**: [FX 트러블슈팅 정리(공개 지식글)](https://infoneva.com/en/knowledge/dealing-with-mitsubishi-fx-plc-error-codes)
- **키워드**: D8060, M8060, 진단, GX Works

#### `ISOLATE-MIN` — 실무 격리 진단법 — 최소 구성 기동

**심각도**: 🔵 절차/지식 · **시리즈**: FX/Q 공통

- **원인**: (절차) 증설/특수 모듈 다수 장착 시 어느 모듈이 원인인지 불명.
- **조치(실무)**: 모든 증설·특수 모듈을 분리하고 베이스(본체)만으로 기동 → 정상이면 모듈을 하나씩 되붙이며 재현 지점을 찾는다. 비호환/구버전 펌웨어 모듈이 문서에 없는 에러를 내는 사례가 보고돼 있다(커뮤니티 검증 절차).
- **근거/출처**: [Oxmaint 커뮤니티 — FX 시리즈 트러블슈팅 스레드](https://www.community.oxmaint.com/discussion-forum/troubleshooting-mitsubishi-fx-plc-melsec-fx2nc-64mt-dss-error-display-and-malfunction-fix)

### 워치독/스캔

#### `6105` — 워치독 타임아웃 — 스캔 200ms(기본) 초과

#### `WDT_ERR` — iQ-R 워치독 오류

## SIEMENS (S7-300/400/1200/1500)

### 통신

#### `OB86` — 랙/스테이션 장애 OB — 분산 I/O 스테이션 전체 상실

**심각도**: 🔴 정지급 · **시리즈**: S7-1500/ET200

#### `BF-LED` — BF(Bus Fault) LED — PROFIBUS/PROFINET 통신 장애

**심각도**: 🟡 경고 · **시리즈**: S7-300/400 DP

- **원인**: DP 주소 중복(1~125 고유해야 함), 종단저항 위치 오류, 슬레이브 전원 다운.
- **조치(실무)**: 점멸=구성된 슬레이브 일부 무응답 / 점등=버스 자체 불가로 갈라 읽는다. 주소맵 표로 중복 검사 → 라인 양끝만 종단 ON 확인 → 슬레이브 전원·커넥터 순회 점검. PN 은 장치명/IP 중복이 같은 증상.
- **근거/출처**: [IndustrialMonitorDirect KB — DP SF/BF 진단](https://industrialmonitordirect.com/blogs/knowledgebase/siemens-s7-300-profibus-sfbf-led-diagnostics-and-configuration)

### I/O·모듈

#### `AI-SF` — 아날로그 입력 모듈 SF — 모듈 전원/결선 문제

**심각도**: 🟡 경고 · **시리즈**: S7-300

#### `F-PASSIVATION` — F-I/O 패시베이션 — 안전 모듈이 페일세이프 값(0)으로 고착

#### `ARRAY-IDX-OOB` — 어레이 인덱스 범위 초과 — 간접 접근이 선언 크기를 벗어남

#### `OB121` — 프로그래밍 오류(동기) — 'DB not loaded'/영역 길이 초과로 STOP

**심각도**: 🔴 정지급 · **시리즈**: S7-300/1500

#### `80E0` — I/O 설정 에러 — 등록 I/O 테이블과 실장 유닛 불일치

**심각도**: 🔴 정지급 · **시리즈**: CJ2

- **원인**: 등록된 I/O 테이블과 실제 접속 유닛이 다름 — 유닛 교체/증설 후 테이블 재생성을 빠뜨린 경우가 전형.
- **조치(실무)**: CX-Programmer 의 I/O 테이블 화면에서 '실장 기반 생성'으로 테이블을 재작성 → 전송 → 전원 재투입. 의도된 구성 변경인지 무단 변경인지 정비 이력과 대조하는 습관이 사고를 줄인다.
- **근거/출처**: [CJ2 CPU 매뉴얼 치명 에러표](https://www.manualslib.com/manual/346472/Omron-Cj2-Cpu-Unit.html?page=367)
- **키워드**: 80E0, I/O 테이블, 불일치, 유닛 교체

#### `80E1` — I/O 점수 초과 — CPU 허용 I/O 포인트 한도 넘김

**심각도**: 🔴 정지급 · **시리즈**: CJ2

- **원인**: 증설을 거듭해 유닛 합계 I/O 점수가 해당 CPU 모델의 최대치를 초과.
- **조치(실무)**: 유닛별 점유 점수를 표로 합산해 CPU 사양 한도와 비교 → 초과분 유닛을 제거하거나 상위 CPU 로 교체를 검토. 증설 계획 단계에서 점수 여유율 20% 이상을 남기는 것이 실무 정석.
- **근거/출처**: [CJ2 CPU 매뉴얼 치명 에러표](https://www.manualslib.com/manual/346472/Omron-Cj2-Cpu-Unit.html?page=367)
- **키워드**: 80E1, I/O 점수, 한도 초과, 증설 계획

#### `80E9` — 유닛/랙 번호 중복 — CPU 버스 유닛 번호가 겹침

**심각도**: 🔴 정지급 · **시리즈**: CJ2

- **원인**: 둘 이상의 CPU 버스 유닛(통신 유닛 등)에 같은 유닛 번호가 할당 — 유닛 전면 로터리 스위치 설정 실수가 대부분.
- **조치(실무)**: 각 CPU 버스 유닛의 번호 스위치를 눈으로 전수 확인해 중복 제거 → 전원 재투입 → I/O 테이블 재생성. 유닛 번호 배정표를 도면에 명기해 두면 교체 작업 때 재발하지 않는다.
- **근거/출처**: [CJ2 CPU 매뉴얼 치명 에러표](https://www.manualslib.com/manual/346472/Omron-Cj2-Cpu-Unit.html?page=367)
- **키워드**: 80E9, 유닛 번호, 중복, 로터리 스위치

#### `IO-BUS-ERR` — I/O 버스 에러 — 증설 랙/커넥터 이상

**심각도**: 🔴 정지급 · **시리즈**: CJ

- **원인**: 증설 케이블 접촉 불량, 랙 전원 저하, 유닛 장착 불량.
- **조치(실무)**: 증설 케이블 재삽입·랙 전원 전압 실측 → 최소 구성 격리법(본체만 기동) 으로 문제 랙 특정.
- **근거/출처**: [OMRON FA 공식](https://www.fa.omron.co.jp/)
- **키워드**: I/O 버스, 증설, 랙

#### `IO_VERIFY_ERR` — NX I/O 슬레이브 검증 오류

**심각도**: 🔴 정지급 · **시리즈**: NX

- **원인**: NX 슬레이브 유닛 구성이 Sysmac Studio 프로젝트와 불일치.
- **조치(실무)**: Sysmac Studio로 실제 구성과 프로젝트를 동기화 후 재전송.
- **키워드**: NX, I/O, slave, verify, Omron, Sysmac

- **키워드**: 배터리, 유지 메모리, PM

#### `BATT_LOW` — CJ 배터리 전압 저하

**심각도**: 🟡 경고 · **시리즈**: CJ

### 프로그램/연산

### CPU/시스템

#### `A400` — 에러코드 저장 체계 — A400(최고 심각도 코드)·에러 로그

**심각도**: 🔵 절차/지식 · **시리즈**: CJ2

- **원인**: (체계 설명) 발생 에러의 코드가 보조 영역 A400 에 저장되고, 동시 다발 시 더 심각한 쪽 코드가 남는다. 상세는 에러별 보조 워드에 추가 기록.

- **근거/출처**: [CJ2 CPU 매뉴얼 에러 종류 장(공개 열람)](https://www.manualslib.com/manual/346472/Omron-Cj2-Cpu-Unit.html?page=202)
- **키워드**: A400, 에러 로그, 보조 영역, 심각도

### 워치독/스캔

#### `809F` — 사이클 타임 초과 — 감시 사이클 시간(watch cycle) 넘김

**심각도**: 🔴 정지급 · **시리즈**: CJ2

- **근거/출처**: [CJ2 CPU 매뉴얼 치명 에러표(공개 열람)](https://www.manualslib.com/manual/346472/Omron-Cj2-Cpu-Unit.html?page=366)
- **키워드**: 809F, 사이클, cycle time, 초과

**심각도**: 🔴 정지급 · **시리즈**: CJ

- **원인**: CJ CPU 사이클 타임이 워치독 설정 이내에 완료되지 않음.

- **키워드**: CJ, watchdog, WDT, cycle, Omron

#### `WDT` — 워치독 타임아웃

**심각도**: 🔴 정지급

- **조치(실무)**: 스캔 부하 분산, 루프 종료 조건 점검, 워치독 설정 확인.
- **키워드**: watchdog, scan overrun, timeout, loop
