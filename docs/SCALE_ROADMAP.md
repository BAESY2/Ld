## 2. 격차 분석 — 대규모로 가기 위해 필요한 것

### A. 시뮬레이션 충실도 (지멘스 PLCSIM Adv. 대비)
- **아날로그 I/O 부재**: 현재 BOOL만. 수위·온도·압력을 INT/REAL(D 레지스터)로, PID 루프 지원 필요 → 엔진 synth에 아날로그 레시피 계열 추가가 선행 조건.
- **스캔 타이밍 정밀도**: 브라우저 rAF(≈16.7ms) 기반. 실PLC 1~10ms 스캔 재현은 서버 코시뮬레이션(아래 C)으로.
- **물리 모델**: 현재 운동학(키네마틱) 수준. 질량/마찰/충돌은 불요 — 로직 검증 목적에는 운동학이 적정. 단, 컨베이어 적체/체적 유량은 보강 여지.

### B. 3D/시각화
- 현재: 캔버스 아이소메트릭(의존성 0) — 데모 최적, 편집 불가.
- 다음: **Three.js(WebGL) 전환** — 카메라 자유 회전/줌, GLTF 설비 라이브러리, 기기 클릭 상세(스크린샷의 RB-101 페이지 방향).
- 최종: **공장 편집기** — 설비 팔레트에서 드래그 배치 → 레시피 자동 매핑 → 사용자 공장 그대로 트윈 생성. 이것이 "대규모 프로젝트"의 핵심 차별화.

### C. 코시뮬레이션 서버 (정밀 검증판)
- 브라우저판은 영업/체험용, **서버판**은 납품용: 기존 `app/simulator.py`(이미 OpenPLC 차분검증 통과)를 WebSocket 스트리밍으로 노출 → 브라우저 3D는 뷰만 담당.
- 효과: 스캔 1ms 정밀도 + Z3 증명 + 타임라인 기록/리플레이 → "검증 패키지" 상품(Stage 2)의 기술 본체.

### D. 실설비 연결 (Stage 3)
- OPC UA 서버/클라이언트 게이트웨이(현재 XGT/MELSEC 직결만 있음) → SCADA/MES 생태계 진입 티켓.
- 실라인 신호를 트윈에 미러링(섀도 모드) → 트윈 예측 vs 실측 차분 알람 = 예지보전 상품.

### E. 레시피 커버리지 32 → 200+
- 우선 확장군: 아날로그 PID(온도/수위/압력), 모션(메카피온 서보 — 이미 motion_home_move 보유), 안전(양수조작·뮤팅 확장), 통신(라인 간 핸드셰이크).
- 커버리지 측정: 실제 현장 래더 코퍼스 대비 합성 가능률을 KPI로(현재 bench 인프라 보유 — `b32a6a8`).

### F. 비전
- 현재: 판정 시뮬(주입률 기반). 다음: 실제 이미지 분류 모델(엣지 GPU) ↔ PLC 인터페이스 표준화(트리거/판정/타임아웃 래더 패턴) — 레시피로 흡수.

### G. 플랫폼/SaaS
- 멀티테넌시(프로젝트/조직), 트윈 상태 영속화(Supabase 등), 협업(설계 리뷰·승인 워크플로), 사용량 과금.
- 보안: 안전커널 정책의 서버 강제(이미 게이트 구조 보유 — `5263572`).

### H. 신뢰·인증 (지멘스와의 최종 격차)
- 단기: 고객 현장 PoC 트랙레코드 축적(검증 리포트 자동 생성 — 이번에 추가된 리포트 내보내기가 씨앗).
- 중기: TÜV 공정 인증 컨설팅 착수, IEC 61508 SIL 인증은 하드와이어 원칙 유지 하에 "로직 보조 도구" 포지션으로 우회 — 인증 비용 없이 가치 전달.

## 4. Goal (정의)

> **"한국어 한 문장 → 수학적으로 증명된 제어 로직 → 브라우저에서 검증된 디지털트윈 → 버튼 한 번에 실라인 배포"**
> 를 지멘스 스택 가격의 1/10, 도입 시간 1/100로 제공하는 것.
> 지멘스가 못 하는 것(자연어 진입·자동 증명·벤더 중립·설치 제로)으로 이기고,
> 지멘스가 잘하는 것(실기 신뢰성)은 트랙레코드와 섀도 모드로 따라잡는다.
